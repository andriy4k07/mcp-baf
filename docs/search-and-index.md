# Пошук по коду та індекс

[← Назад до змісту](README.md)

`search_code` — єдиний інструмент, який працює не з живою базою, а з **локальною
вивантаженням конфігурації** (`DumpConfigToFiles`). Під ним лежить повнотекстовий
індекс SQLite FTS5 над `.bsl`-файлами вивантаження.

> **Без `--dump` тула немає взагалі.** Якщо шлях до вивантаження не заданий
> (ні прапором `--dump`, ні змінною `mcp_baf_DUMP_DIR`), `search_code` не
> реєструється в MCP-сервері — він не з'явиться у списку інструментів клієнта.
> Решта інструментів працюють як звичайно. Див. [configuration.md](configuration.md).

Порт `dump/index.go` з Go-версії; пошуковий рушій замінено з bleve на вбудований
у Python SQLite FTS5, тому зовнішніх залежностей для пошуку немає.

## Як отримати вивантаження

Конфігуратор → **Конфігурація** → **Вивантажити конфігурацію у файли**, або з
командного рядка:

```sh
1cv8 DESIGNER /F "шлях/до/бази" /DumpConfigToFiles "шлях/до/вивантаження"
```

Далі шлях передається серверу:

```sh
mcp-baf --base http://localhost:8080/hs/mcp-baf --dump "шлях/до/вивантаження"
```

Вивантаження — **знімок**, а не жива база: після зміни коду в Конфігураторі його
треба перевивантажити, інакше `search_code` шукатиме по застарілих модулях.
Індекс сам помітить змінені файли при наступному старті (див. нижче).

## Неблокуючий старт

Індекс будується у **фоновому потоці** — MCP-сервер стартує одразу і решта
інструментів доступні негайно. Поки індекс не готовий, `search_code` не мовчить
і не блокується, а повертає зрозумілу помилку:

```
search index is building, please retry
```

Це навмисна поведінка (як у Go-версії): AI-клієнт отримує явну відповідь і може
повторити запит, замість того щоб чекати на таймаут при старті великої бази.

Готовність індексу — `threading.Event`; доступ до SQLite серіалізовано окремим
`threading.Lock`, бо з'єднання відкрите з `check_same_thread=False` і його ділять
фоновий будівник та потік обробки запитів.

## Дисковий кеш

База індексу зберігається на диску, тому повторний старт майже миттєвий.

Каталог кеша — **перші 16 hex-символів sha256** від абсолютного шляху вивантаження
(`dumpindex/cache.py`), усередині файл `index.db`:

| ОС | Шлях |
|---|---|
| Windows | `%LocalAppData%/mcp-baf/<hash>/index.db` |
| Linux | `~/.cache/mcp-baf/<hash>/index.db` (або `$XDG_CACHE_HOME`) |
| macOS | `~/Library/Caches/mcp-baf/<hash>/index.db` |

Базовий каталог перевизначається через `--cache-dir` / `mcp_baf_CACHE_DIR`. Різні
вивантаження не конфліктують — у кожної свій хеш.

### Інкрементальне оновлення

`mtime_ms` і `size` кожного файла зберігаються прямо в таблиці `docs` і разом
утворюють маніфест. При старті кеш відкривається і звіряється з диском: заново
читаються тільки ті модулі, у яких змінився час модифікації або розмір. Решта
береться з кеша як є.

`--reindex` ігнорує кеш і будує індекс з нуля.

### Версія схеми

`SCHEMA_VERSION = 2` (`dumpindex/index.py`). При розбіжності кеш **відкидається
й будується заново** — без ручного втручання. Версію піднято до 2 разом з
NFC-нормалізацією імен модулів: кеші, зібрані до того фіксу на macOS, містять
NFD-ключі, які NFC-запит ніколи не знайде.

## Нормалізація імен на macOS

macOS розпаковує вивантаження з кириличними іменами в **NFD**: `Й` і `Ё` (в обох
регістрах) лежать на диску як база + комбінований знак, тоді як запити й імена з
HTTP-відповідей 1С приходять у **NFC**. Без нормалізації пошук по модулю з `Й` в
імені просто не знаходив би нічого — байти різні при візуально однаковому тексті.

`dumpindex/modulenames.py:nfc()` — єдина точка нормалізації: усі ключі, виведені
зі шляхів на диску, проходять через неї. Уже нормалізований вхід (Windows, Linux,
HTTP) повертається без змін. Покрито `tests/test_modulenames.py`.

## Режими пошуку

Параметр `mode`; типово `smart`. `limit` — типово 50, стеля 500 (значення ≤ 0
замінюється дефолтом).

### `smart` — BM25 + синоніми

Запит розбивається на токени регуляркою `[^\W_]+` — так само сегментує
ідентифікатори токенізатор `unicode61` самого FTS5, тому підкреслення межує
токени. Кожен токен, у якого є BSL-синонім, розгортається в пару:

```
СтрНайти  →  ("стрнайти" OR "strfind")
```

Токени об'єднуються через `AND`, ранжування — `bm25(fts)` (FTS5 із зовнішнім
вмістом `content='docs'`, тож індекс не дублює тексти модулів). Результат
повертається зі `score`.

Саме тому пошук за англійською назвою знаходить російський код і навпаки —
розгортання відбувається **на етапі запиту**, а не при індексації, тому зміна
таблиці синонімів не потребує перебудови індексу.

### `regex` — регулярний вираз

Побудковий скан вмісту модулів через `re` Python. **Синтаксис відрізняється** від
Go `regexp` оригіналу — це різні рушії. Некоректний вираз піднімає
`invalid regex ...` замість тихого нуля результатів.

### `exact` — точний підрядок

Побудковий скан, підрядок без урахування регістру. Ні синонімів, ні ранжування.

`smart` шукає по індексу, `regex`/`exact` — скан по вмісту, тож на великих
вивантаженнях вони помітно повільніші.

## Синоніми BSL

`dumpindex/synonyms.py` будує **двонаправлену** карту (`build_synonym_map`,
кешується `lru_cache`):

- **53 пари ключових слів** мови — `процедура`↔`procedure`, `если`↔`if`,
  `выбрать`↔`select`, `количество`↔`count` і т.д.
- **Вбудовані функції платформи** з `mcp_baf.bsl.BUILTIN_FUNCTIONS` — пара
  `name` ↔ `name_en`.

Пара функції пропускається цілком, якщо будь-який з її ключів уже зайнятий —
інакше двонаправлені ланцюжки ламаються, коли ключове слово й функція ділять одне
англійське ім'я. Пари, де обидві сторони збігаються, теж пропускаються.

## Фільтри

Обидва фільтри **чутливі до регістру** — вони звіряються з полями `docs.category`
і `docs.module` через точну рівність, тож `документ` не знайде нічого, а
`Документ` знайде:

| Параметр | Приклад | Що фільтрує |
|---|---|---|
| `category` | `Документ`, `Справочник`, `ОбщийМодуль` | Тип метаданих |
| `module` | `МодульОбъекта`, `МодульМенеджера` | Тип модуля |

Порожній рядок означає «без фільтра».

## Результат

Кожне влучання — модуль, номер рядка, контекст (**±2 рядки** навколо знайденого,
`CONTEXT_WINDOW = 2`) і `score` (лише в `smart`). Разом з влучаннями повертається
загальна кількість збігів, тому видно, чи результат обрізано по `limit`.

Для `smart` номер рядка шукається спершу по вихідних токенах запиту; якщо FTS5
знайшов документ **через синонім**, а вихідні токени в тексті не зустрічаються —
пошук повторюється по розширеному набору токенів. Якщо не знайдено й так, береться
перший рядок.

## Продуктивність

Модулі читаються паралельно (`ThreadPoolExecutor`) — навантаження впирається в
диск, тому потоків достатньо.

> README називає орієнтир «~2 сек на 5 000 модулів». Це заявлений орієнтир, а не
> зафіксований у коді чи тестах бенчмарк — фактичний час залежить від диска,
> розміру вивантаження й того, чи є кеш.

## Порожнє вивантаження

Якщо в каталозі не знайдено жодного `.bsl`, `search_code` повертає текстове
пояснення, а не помилку — типова причина в тому, що вказано не той каталог
(наприклад корінь проєкту замість каталогу вивантаження).
