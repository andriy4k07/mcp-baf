# Конфігурація

[← Назад до README](../README.md)

Усі параметри сервера задаються прапором CLI **або** змінною оточення. Прапори
поділяються на дві групи:

- **параметри роботи сервера** — впливають на запущений MCP-сервер
  (підключення до 1С, кеш/логи/аудит, пошуковий індекс, прогрес у stderr);
- **параметри режиму встановлення** (`--install …`) — використовуються лише
  одноразово під час розгортання розширення в базу 1С і на роботу сервера не
  впливають (детальніше — у [Швидкому старті](getting-started.md)).

## Як задаються параметри

**Пріоритет джерел** (від нижчого до вищого):

```
значення за замовчуванням  →  змінна оточення  →  прапор CLI
```

Тобто прапор CLI перекриває змінну оточення, а та — дефолт. Логіка зібрана в
`load_config()` (`src/mcp_baf/config.py`): спершу `Config` заповнюється зі
значень оточення (або дефолтів, якщо змінної немає), потім непорожні/додатні
значення з `argparse.Namespace` перезаписують поля.

> **Регістр імені env-змінних.** Префікс — саме **малими літерами**
> `mcp_baf_` (наприклад `mcp_baf_BASE_URL`), а не `MCP_BAF_` — так написано в
> коді (`os.environ.get("mcp_baf_BASE_URL", ...)`), і на більшості систем це
> має значення (Linux/macOS чутливі до регістру). Дотримуйтесь точного
> написання.

**Кілька правил парсингу, про які корисно знати:**

- **Числові параметри** (`--max-response-size`, `--request-timeout`,
  `--audit-max-size`, `--audit-archives`, а також їхні env-відповідники)
  приймають лише **додатне ціле**. У коді це реалізовано подвійним запобіжником:
  - CLI-прапори оголошені з `type=int, default=0` — значення застосовується,
    лише якщо `> 0`. Отже `--max-response-size 0` **нічого не змінює**: нуль
    трактується як «не задано», а не як спроба вимкнути ліміт.
  - Env-змінні читає хелпер `_env_int()`: некоректне (не число) **або**
    недодатне значення (зокрема `0` і від'ємні) мовчки ігнорується — лишається
    дефолт. Поведінка навмисно повторює Go-версію.
- **Пароль** (`--pass`/`--password`, `--db-password`) у журналі аудиту
  автоматично редагується до `***` (пакет `mcp-baf-audit`, див.
  [Логування та аудит](logging-and-audit.md)) — сам пароль у файлах не
  зберігається у відкритому вигляді.
- `--reindex`, `--debug`, `--quiet`, `--verbose` задаються **тільки** прапором
  CLI — окремої змінної оточення саме під ці прапори немає (у `--quiet` є
  окремий, не однойменний, env-перемикач `mcp_baf_NO_TTY` — див. нижче).
- Усі прапори режиму встановлення (`--install`, `--lang`, `--server`,
  `--platform`, `--platform-version`, `--db-user`, `--db-password`) —
  **тільки CLI**, без env.

## Параметри роботи сервера

| Прапор | Env var | За замовчуванням | Опис |
|--------|---------|------------------|------|
| `--base` | `mcp_baf_BASE_URL` | `http://localhost:8080/hs/mcp-baf` | URL HTTP-сервісу 1С |
| `--user` | `mcp_baf_USER` | — | Користувач HTTP-сервісу 1С |
| `--pass` / `--password` | `mcp_baf_PASSWORD` | — | Пароль HTTP-сервісу 1С |
| `--max-response-size` | `mcp_baf_MAX_RESPONSE_SIZE` | `128` | Максимальний розмір відповіді 1С у мебібайтах (MiB) |
| `--request-timeout` | `mcp_baf_REQUEST_TIMEOUT` | `300` | Таймаут HTTP-запиту до 1С у секундах |
| `--dump` | `mcp_baf_DUMP_DIR` | — (порожньо) | Шлях до вивантаження конфігурації (`DumpConfigToFiles`); вмикає `search_code` і збагачення `get_form_structure` |
| `--cache-dir` | `mcp_baf_CACHE_DIR` | платформений кеш (див. нижче) | Каталог для `audit.log`, `server.log` і кешу пошукового індексу |
| `--reindex` | — | вимкнено | Примусова перебудова кешу пошукового індексу |
| `--debug` | — | вимкнено | DEBUG-рівень у `server.log` + подробиці httpx; також глушить прогрес у stderr |
| `--quiet` | — (є `mcp_baf_NO_TTY`, див. нижче) | вимкнено | Примусово заглушити прогрес навіть у терміналі; перекриває `--verbose` |
| `--verbose` | — | вимкнено | Примусово ввімкнути прогрес навіть при запуску через pipe |
| `--audit-max-size` | `mcp_baf_AUDIT_MAX_SIZE` | `50` | Ротація `audit.log` при перевищенні розміру (MiB) |
| `--audit-archives` | `mcp_baf_AUDIT_ARCHIVES` | `20` | Кількість архівів `audit-*.log`, що зберігаються |
| `--version` | — | — | Вивести версію `mcp-baf` (`mcp-baf <version>`) і вийти |

Далі — детально по групах.

### Підключення до 1С

- **`--base` / `mcp_baf_BASE_URL`** (типово
  `http://localhost:8080/hs/mcp-baf`) — повний URL HTTP-сервісу, який реєструє
  розширення в базі. Значення має збігатися з реальним шляхом публікації:
  - вбудований HTTP-сервер `/HTTPPort`: `http://localhost:8080/hs/mcp-baf`;
  - публікація через Apache/IIS: `http://host/<публікація>/hs/mcp-baf`.

  > Сегмент `/hs/mcp-baf` зашитий у розширенні (`RootURL` HTTP-сервісу
  > `MCPService`) і змінювати його не можна — змінюється лише частина URL до
  > нього.

- **`--user` / `--pass`** (`mcp_baf_USER` / `mcp_baf_PASSWORD`) — обліковий
  запис 1С для базової автентифікації HTTP-сервісу. Якщо `--user` не заданий —
  запити йдуть без автентифікації (`OneCClient` додає `httpx.BasicAuth`, лише
  якщо `config.user` непорожній).

- **`--max-response-size`** (типово **128 MiB**) — верхня межа розміру
  відповіді 1С. Клієнт (`client.py`) читає відповідь потоково і рахує байти;
  щойно розмір перевищив ліміт, читання перериває з `OneCError` замість того,
  щоб чекати повного завантаження і ризикувати OOM. У коментарі до дефолту
  пояснення: 128 MiB покриває великі бази з розширеннями, лишаючись при цьому
  розумною стелею проти OOM. Помилка при перевищенні прямо підказує, який
  прапор/env підняти.

- **`--request-timeout`** (типово **300 секунд**) — таймаут одного HTTP-запиту
  до 1С (`httpx.Timeout`). Дефолт навмисно високий: коментар у коді пояснює,
  що запас потрібен для передачі великих відповідей `/extensions` (сотні
  мегабайт). Піднімайте, якщо запити до великої бази впираються в таймаут.

- Клієнт завжди шле заголовок `Connection: close` — це навмисно, щоб не
  вичерпувати ліміт сеансів 1С численними keep-alive з'єднаннями; на
  конфігурацію не впливає, але пояснює, чому кожен виклик інструмента — це
  окреме TCP-з'єднання.

### Пошуковий індекс (`--dump`)

- **`--dump` / `mcp_baf_DUMP_DIR`** — шлях до каталогу вивантаження
  конфігурації (`DumpConfigToFiles`). Якщо не заданий (порожній рядок,
  дефолт) — інструмент `search_code` **взагалі не реєструється** в
  MCP-сервері (перевіряється тестами: `server.py` реєструє `search_code.register`
  лише за умови `if index is not None`, а `index` створюється лише коли
  `config.dump_dir` непорожній). З `--dump` також збагачується
  `get_form_structure` — дані елементів/команд/обробників форми беруться з
  `Form.xml` у вивантаженні.
- **`--reindex`** — форсує повну перебудову кешу індексу навіть якщо на диску
  вже є актуальний. Без нього кеш оновлюється інкрементально за mtime+size
  змінених файлів.

  Детальніше про сам механізм індексації — у
  [Пошуку по коду та індексі](search-and-index.md).

### Кеш, логи та аудит

- **`--cache-dir` / `mcp_baf_CACHE_DIR`** — каталог, куди пишуться
  `audit.log`, `server.log` і кеш пошукового індексу (у підкаталозі, ім'я
  якого — перші 16 hex-символів sha256 від абсолютного шляху `--dump`). Якщо
  не заданий — використовується платформений кеш-каталог з іменем застосунку
  `mcp-baf`:
  - Windows: `%LOCALAPPDATA%\mcp-baf\`
  - macOS: `~/Library/Caches/mcp-baf/`
  - Linux: `~/.cache/mcp-baf/` (або `$XDG_CACHE_HOME/mcp-baf/`, якщо
    змінна задана)

  Цей самий дефолт незалежно обчислюють `__main__.py` (для `server.log`,
  через `default_cache_dir("mcp-baf")` з `mcp-baf-audit`) і `AuditLog` (для
  `audit.log`, той самий хелпер з `service="mcp-baf"`) — тому обидва журнали
  завжди лежать поруч, навіть якщо `--cache-dir` не задано.

- **`--audit-max-size`** (типово **50 MiB**) — поріг ротації `audit.log`: при
  перевищенні файл перейменовується в `audit-<timestamp>.log`.
- **`--audit-archives`** (типово **20**) — скільки архівів `audit-*.log`
  зберігати; старші за ліміт видаляються.
- **`--debug`** — піднімає рівень кореневого логера до DEBUG (лишає
  `server.log` детальним, вмикає подробиці `httpx`/`httpcore`, які за
  замовчуванням прицмикнуті до WARNING). У stderr незалежно від `--debug`
  потрапляють лише повідомлення рівня ERROR — MCP-клієнти показують кожен
  рядок stderr як помилку. Побічний ефект `--debug`: він також вимикає
  індикатор прогресу в stderr (див. нижче).

Деталі формату журналів, схеми подій і ротації — у
[Логуванні та аудиті](logging-and-audit.md).

### Прогрес індексації на stderr

Індексація `search_code` виводить прогрес (наприклад «проіндексовано N
модулів») у stderr рядками виду `[HH:MM:SS] повідомлення`. Кому це показувати
— вирішує невелика логіка в `load_config()`:

1. Базове значення — **чи stdin є терміналом**: `show = sys.stdin.isatty()`.
   При звичайному запуску AI-клієнтом (stdio-підпроцес) stdin — це pipe, тож
   за замовчуванням прогрес **вимкнено**; при ручному запуску в терміналі —
   **увімкнено**.
2. **`--verbose`** форсує `show = True` незалежно від TTY — зручно для
   діагностики підключення MCP-клієнта, коли треба побачити прогрес навіть
   через pipe.
3. **`--quiet`** АБО **`mcp_baf_NO_TTY=1`** форсують `show = False` —
   перекриває навіть `--verbose`, якщо обидва задані одночасно.
4. **`--debug`** застосовується останнім і завжди вимикає індикатор
   (`cfg.show_progress = show and not args.debug`) — у режимі debug весь вивід
   іде у файл `server.log`, а не в stderr, незалежно від попередніх трьох
   пунктів.

> **Git Bash / MSYS2 / MinTTY на Windows:** ці оболонки підключають stdin
> через іменовані pipes, тому автовизначення вважає їх не-TTY і прогрес не
> виводиться. Для ручної діагностики використовуйте `--verbose` або
> повноцінний `cmd.exe` / Windows Terminal.

## Параметри режиму встановлення (`--install`)

Ці прапори діють **тільки разом із `--install`** і потрібні одноразово під
час розгортання розширення в базу (детальніше, з прикладами команд і
розбором помилок — у [Швидкому старті](getting-started.md)). Змінних
оточення вони не мають.

| Прапор | Опис |
|--------|------|
| `--install DB_PATH` | Встановити розширення `MCPService` у базу за вказаним шляхом (файлова база) або рядком `сервер\база` (з `--server`) |
| `--server` | Трактувати значення `--install` як рядок підключення до клієнт-серверної бази (`сервер\база`) |
| `--lang {ua,ru}` | Мова синонімів об'єктів розширення: `ua` (українська, типово) або `ru` (російська) |
| `--platform` | Шлях до бінарника платформи 1С (автовизначення, якщо не вказано) |
| `--platform-version` | Версія платформи (напр. `8.3.13`), якщо її не видно зі шляху. Мінімальна підтримувана версія — **8.3.10**; на старіших `--install` завершується помилкою ще до звернення до DESIGNER |
| `--db-user` | Користувач бази 1С для DESIGNER (Конфігуратора) |
| `--db-password` | Пароль бази 1С для DESIGNER |

## Приклади

**Мінімум — лише підключення до 1С (HTTP-сервіс без автентифікації):**

```json
{
  "mcpServers": {
    "mcp-baf": {
      "command": "/шлях/до/.venv/bin/mcp-baf",
      "args": ["--base", "http://localhost:8080/hs/mcp-baf"]
    }
  }
}
```

**Усе через змінні оточення (секрети не потрапляють у список процесів):**

```json
{
  "mcpServers": {
    "mcp-baf": {
      "command": "/шлях/до/.venv/bin/mcp-baf",
      "env": {
        "mcp_baf_BASE_URL": "http://host/INTEGRA/uk_UA/hs/mcp-baf",
        "mcp_baf_USER": "Адміністратор",
        "mcp_baf_PASSWORD": "пароль",
        "mcp_baf_DUMP_DIR": "/шлях/до/вивантаження",
        "mcp_baf_CACHE_DIR": "/шлях/до/кешу"
      }
    }
  }
}
```

**Через CLI-прапори, з увімкненим `search_code` і піднятими лімітами для
великої бази з розширеннями:**

```json
{
  "mcpServers": {
    "mcp-baf": {
      "command": "C:\\шлях\\до\\.venv\\Scripts\\mcp-baf.exe",
      "args": [
        "--base", "http://host/INTEGRA/uk_UA/hs/mcp-baf",
        "--dump", "D:\\1c\\dump",
        "--max-response-size", "256",
        "--request-timeout", "600"
      ],
      "env": {
        "mcp_baf_USER": "Адміністратор",
        "mcp_baf_PASSWORD": "пароль"
      }
    }
  }
}
```

**Заглушення прогресу в Docker/systemd (без TTY і без `--debug`):**

```json
{
  "mcpServers": {
    "mcp-baf": {
      "command": "mcp-baf",
      "args": ["--base", "http://localhost:8080/hs/mcp-baf"],
      "env": {
        "mcp_baf_NO_TTY": "1"
      }
    }
  }
}
```
