# Перелік інструментів

[← Назад до README](README.md)

`mcp-baf` — read-only MCP-сервер: усі інструменти лише читають дані 1С
(`annotations=ToolAnnotations(readOnlyHint=True)` на кожному) і не мають
побічних ефектів. Кожен тул повертає готовий **markdown**-текст, а не JSON —
форматування відповіді (заголовки, таблиці, списки) робить сам тул
(`tools/*.py:format_*`), MCP-клієнт показує його як є.

Порядок реєстрації в `server.py:create_server` навмисно збігається з
Go-версією (`server/server.go`):

```
get_metadata_tree → get_object_structure → execute_query →
search_code (лише якщо задано --dump) → get_form_structure →
validate_query → get_event_log → get_configuration_info → bsl_syntax_help
```

Кожен виклик обгорнутий у `tools/common.py:traced_text`: пише подію аудиту
`tool_call` (успіх, `duration_ms`, `args`) або `tool_error` (виняток
перевикидається далі) і закріплює `trace_id` в contextvar — вкладені
`one_c.http`-події `OneCClient` успадковують той самий `trace_id`.

## Інструменти

| Інструмент | Призначення, параметри, обмеження | 1С / джерело |
|---|---|---|
| `get_metadata_tree` | Дерево метаданих конфігурації: довідники, документи, регістри, переліки, обробки, звіти, загальні модулі та ін. Без параметра `filter` — компактна зведена таблиця (категорія + кількість об'єктів, з підказкою `filter="Категорія"`). З `filter` (рядок, точна назва категорії 1С, напр. `Справочники`, `Документы`, `РегистрыСведений`, `ОбщиеМодули`) — повний перелік об'єктів категорії. Автогенеровані об'єкти з суфіксом `ПрисоединенныеФайлы`/`ПрисоединённыеФайлы` відфільтровуються як шум. Невідомі категорії з відповіді 1С дописуються в кінець (forward compatibility) | GET `/metadata` |
| `get_object_structure` | Реквізити, табличні частини, виміри, ресурси і типи полів об'єкта метаданих. Для `object_type=Enum` додатково повертається секція **Значення** зі списком значень перелічення. Обидва параметри обов'язкові: `object_type` (`Catalog`, `Document`, `Enum`, `InformationRegister`, `AccumulationRegister`, `AccountingRegister`, `CalculationRegister`, `ChartOfAccounts`, `ChartOfCharacteristicTypes`, `ChartOfCalculationTypes`, `ExchangePlan`, `BusinessProcess`, `Task`, `DataProcessor`, `Report`), `object_name` (ім'я об'єкта, напр. `РеализацияТоваровУслуг`) | GET `/object/{object_type}/{object_name}` |
| `execute_query` | Виконати запит мовою запитів 1С і отримати дані (елементи довідника, документи, залишки, обороти тощо). **Лише SELECT/ВЫБРАТЬ**: перші 30 символів запиту (`.strip().upper()`) мусять починатися з `ВЫБРАТЬ` або `SELECT`, інакше `ValueError` до звернення в 1С; розширення 1С додатково саме гарантує read-only виконання на сервері. Параметри: `query` (обов'язковий, підтримує `&Имя`-параметри), `limit` (типово 100, стеля 1000, значення ≤0 замінюється дефолтом), `parameters` (словник ключ→значення, ключ без амперсанда). Відповідь — markdown-таблиця (`columns`/`rows`), позначка `truncated`, якщо результат обрізано | POST `/query` |
| `search_code` | **Реєструється лише якщо задано `--dump`/`mcp_baf_DUMP_DIR`** — без dump-каталогу цей тул взагалі відсутній у списку MCP-тулів (`server.py`: `if index is not None: search_code.register(...)`). Повнотекстовий пошук по коду модулів вивантаження. Три режими (`mode`): `smart` (типово, BM25 + BSL-синоніми ru↔en), `regex` (Python `re`, синтаксис відрізняється від Go regexp оригіналу), `exact` (підрядок без урахування регістру). Параметри: `query` (обов'язковий), `limit` (типово 50, стеля 500), `category` (тип метаданих, **чутливо до регістру**, напр. `Документ`, не `документ`), `module` (тип модуля, **чутливо до регістру**, напр. `МодульОбъекта`). До готовності індексу піднімає помилку `search index is building, please retry`; на порожньому dump-каталозі (0 знайдених `.bsl`) повертає текстове пояснення замість помилки. Деталі — [search-and-index.md](search-and-index.md) | Локальний SQLite FTS5-індекс (не жива 1С) |
| `get_form_structure` | Структура керованої форми: елементи, команди, обробники подій. HTTP-ендпоінт 1С у серверному контексті **не віддає** склад елементів і обробники (нема доступу до `ФормаКлиентскогоПриложения`) — тільки ім'я і заголовок (синонім із конфігурації). Повний склад підтягується з `Form.xml` локальної dump-вивантаження, коли сервер запущений з `--dump`; **без dump тул усе одно зареєстрований**, просто повертає лише ім'я/заголовок. Обов'язкові `object_type`, `object_name`; необов'язковий `form_name` — без нього береться перша форма за алфавітом серед знайдених у dump. Якщо обидва джерела (HTTP і dump) впали — комбінована помилка з текстом обох причин | GET `/form/{object_type}/{object_name}` + `Form.xml` з dump (якщо задано) |
| `validate_query` | Перевірка синтаксису запиту 1С **без виконання**. Один обов'язковий параметр `query`. Відповідь — ✅ «Запрос корректен» або ❌ список помилок з відповіді 1С | POST `/validate-query` |
| `get_event_log` | Читання журналу реєстрації 1С з фільтрацією. Параметри (усі необов'язкові, крім жодного): `start_date`/`end_date` (ISO 8601, напр. `2026-03-01T00:00:00`), `level` (`Ошибка` \| `Предупреждение` \| `Информация` \| `Примечание`), `user` (ім'я користувача 1С), `limit` (типово 50, стеля 500). Кожен запис виводиться з датою, рівнем, подією, користувачем і за наявності — комп'ютером/метаданими/даними/коментарем/транзакцією | POST `/eventlog` |
| `get_configuration_info` | Загальна інформація про базу: назва конфігурації, версія, постачальник, версія платформи, режим роботи (`file` → «Файловый», `server` → «Клиент-серверный»). Без параметрів | GET `/configuration` |
| `bsl_syntax_help` | Довідка по 180 вбудованих функціях, методах типів (`ТаблицаЗначений`, `Массив`, `Структура`, `Соответствие`, `Запрос` та ін.) і патернах BSL. Працює **повністю локально** — дані з `mcp_baf.bsl._functions_data` (згенеровані з Go-репо скриптом `scripts/gen_bsl_data.py`), без HTTP-запитів і без запущеної 1С. Один параметр `query` — назва функції рос./англ., пошук підрядком без урахування регістру (`СтрНайти` і `StrFind` знаходять один запис). Порожній результат — текстове повідомлення «не знайдена», не помилка | Локальні дані пакета (без 1С) |

> Формально «9 інструментів для роботи з живою базою» (README) — неточно для
> двох із них: `search_code` працює по локальному dump-індексу (не по живій
> базі, і взагалі не реєструється без `--dump`), а `bsl_syntax_help` не
> звертається до 1С зовсім. Решта сім справді йдуть в HTTP-сервіс 1С,
> `get_form_structure` — комбіновано (HTTP + опційно dump).

## Промпти (11)

Крім тулів сервер реєструє 11 MCP-промптів (`prompts.py:register`, порт
`prompts/prompts.go` з Go-версії) — готові покрокові інструкції для типових
задач розробки 1С, що спираються на тули вище. Кожен промпт повертає одне
користувацьке повідомлення (звичайний текст із планом дій), MCP-клієнт
підставляє його в діалог з моделлю.

| Промпт | Аргументи | Що дає |
|---|---|---|
| `review_module` | `object_type`, `object_name` | Ревʼю коду модуля об'єкта: помилки, порушення стандартів 1С, продуктивність (запити в циклі), читаність (зарезервовані слова як імена), робота з транзакціями/блокуваннями |
| `write_posting` | `document_name` | Допомога з написанням `ОбработкаПроведения(Отказ, РежимПроведения)`: рухи по регістрах, запит з’єднань по табличній частині, контроль залишків в оперативному режимі |
| `optimize_query` | `query` | Аналіз і оптимізація запиту: `LEFT`/`INNER JOIN`, індекси, віртуальні таблиці з параметрами замість вкладених запитів, `РАЗЛИЧНЫЕ`/`ПЕРВЫЕ`/`СГРУППИРОВАТЬ`, пакетні запити |
| `explain_config` | — | Пояснення структури поточної конфігурації: підсистеми, довідники, документи, регістри, плани рахунків, звіти/обробки, ролі, архітектура предметної області |
| `analyze_error` | `error_text` | Визначення типу помилки 1С (синтаксична / часу виконання / запиту / блокування / прав доступу), причина, спосіб виправлення, профілактика |
| `find_duplicates` | `object_type`, `object_name` | Пошук дублювального/надлишкового коду в модулях об'єкта: copy-paste, схожі процедури, повторювані запити, кандидати на винесення в загальний модуль |
| `write_report` | `description` | Допомога з написанням звіту: текст запиту для СКД, структура налаштувань (групування, поля, відбори, умовне оформлення), код модуля звіту |
| `explain_object` | `object_type`, `object_name` | Пояснення призначення й устрою об'єкта: реквізити, табличні частини, зв'язки з іншими об'єктами (посилальні типи), бізнес-логіка модулів |
| `1c_query_syntax` | — | Шпаргалка синтаксису запитів 1С: іменування таблиць в однині, віртуальні таблиці регістрів, робота з переліками через `ЗНАЧЕНИЕ()`, типові помилки з прикладами |
| `1c_metadata_navigation` | — | Порядок дослідження незнайомої конфігурації (`get_configuration_info` → `get_metadata_tree` без фільтра → з фільтром → `get_object_structure`), маппінг категорій метаданих на імена таблиц запитів |
| `1c_development_workflow` | `task` | Покроковий workflow розробки під конкретну задачу + чекліст перед написанням коду + повний список зарезервованих слів BSL (рос./англ.) |
