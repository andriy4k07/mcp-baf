# mcp-baf

MCP-сервер для інтеграції AI-асистентів з BAF: Малий Бізнес/1С:УНФ.

AI бачить метадані вашої конфігурації 1С і генерує точний код на BSL. Працює з будь-яким MCP-сумісним клієнтом.

Python-порт [feenlace/mcp-1c](https://github.com/feenlace/mcp-1c) (Go, MIT): ті самі інструменти, промпти, пошук по коду з дисковим кешем і режим установки розширення.

### Працює з локальними моделями

mcp-baf не прив'язаний до конкретної нейромережі. Працює з будь-яким MCP-сумісним клієнтом:

- **Локальні моделі** (Ollama, LM Studio, llama.cpp) — дані не покидають вашу мережу
- **Хмарні сервіси** (Claude, GPT та інші) — через відповідні MCP-клієнти
- **IDE з AI** (Cursor, VS Code + Continue/Cline, JetBrains)

Ваш код і дані 1С залишаються у вас. mcp-baf — це локальний процес, який спілкується тільки з вашою базою.

## Чому mcp-baf

- **9 інструментів для роботи з живою базою.** Метадані, інформація про конфігурацію, форми, запити до даних (з параметрами), пошук по коду, валідація запитів, журнал реєстрації, довідка BSL.
- **11 промптів** для типових задач розробки: рев'ю модуля, обробка проведення, оптимізація запиту, аналіз помилки, пошук дублів та інші.
- **Повнотекстовий пошук по коду** (`search_code`). Три режими: smart (BM25-ранжування), regex, exact. Вбудовані BSL-синоніми — пошук за `StrFind` знаходить `СтрНайти` і навпаки.
- **Швидка індексація з дисковим кешем.** Паралельне читання модулів, ~2 сек на 5 000 модулів. Кеш на диску з інкрементальним оновленням — повторний запуск миттєвий, змінені файли переіндексовуються за mtime+size.
- **Неблокуючий старт.** Індекс будується у фоновому потоці, MCP-сервер доступний одразу. Пошук запрацює після завершення індексації.
- **Працює з вашою базою.** AI бачить реальну конфігурацію і реальні дані — не абстрактну довідку, а саме вашу базу.
- **Не прив'язаний до IDE і нейромережі.** Працює з Конфігуратором, EDT або взагалі без IDE. Працює з будь-якою моделлю, включно з локальними (Ollama, LM Studio). Потрібен лише HTTP-сервіс 1С.
- **Автоустановка розширення.** `mcp-baf --install "C:\шлях\до\бази"` — сам знайде платформу, поставить розширення, оновить конфігурацію БД.
- **Вбудована довідка BSL.** Синтаксис 180 функцій платформи доступний без зовнішніх сервісів і без запущеної 1С.
- **Чистий Python.** Офіційний MCP SDK + httpx + стандартна бібліотека (SQLite FTS5 для пошуку). Потрібен лише Python 3.11+.

## Швидкий старт

### 1. Встановити

```sh
git clone https://github.com/andriy4k07/mcp-baf
cd mcp-baf
python -m venv .venv
.venv\Scripts\pip install -e .      # Windows
# .venv/bin/pip install -e .        # Linux / macOS
```

### 2. Встановити розширення в 1С

```sh
# Windows
mcp-baf --install "C:\шлях\до\бази"

# macOS / Linux
mcp-baf --install ~/Documents/InfoBase

# Клієнт-серверна база (MS SQL, PostgreSQL)
mcp-baf --install "srv-1c\buh_prod" --server --db-user Admin --db-password pass

# Російські синоніми об'єктів розширення (за замовчуванням — українські)
mcp-baf --install "C:\шлях\до\бази" --lang ru
```

> Якщо платформа встановлена в нестандартну папку: `mcp-baf --install "шлях" --platform "/custom/path/to/1cv8"`
>
> Якщо версія платформи не визначається автоматично (нестандартний шлях без номера версії), вкажіть її явно: `--platform-version 8.3.13`. Мінімальна підтримувана версія: **8.3.10**.

### 3. Запустити HTTP-сервіс 1С

**Рекомендований спосіб** — стандартна публікація через Apache або IIS (Конфігуратор → Адміністрування → Публікація на веб-сервері). Працює на Windows і Linux.

**Швидкий запуск для розробки (тільки Windows):**

```cmd
"C:\Program Files\BAF\8.3.XX.XXXX\bin\1cv8.exe" ENTERPRISE /F "C:\шлях\до\бази" /HTTPPort 8080
```

> Параметр `/HTTPPort` — недокументований, працює **тільки на Windows** і підходить для швидкої перевірки. На Linux використовуйте Apache або ibsrv, на macOS HTTP-сервіси 1С не підтримуються.

### 4. Налаштувати AI-клієнт

Конфігурація MCP-сервера однакова для будь-якого клієнта і будь-якої моделі:

```json
{
  "mcpServers": {
    "mcp-baf": {
      "command": "/шлях/до/.venv/bin/mcp-baf",
      "args": ["--base", "http://localhost:8080/hs/mcp-baf"],
      "env": {
        "mcp_baf_USER": "Адміністратор",
        "mcp_baf_PASSWORD": "пароль"
      }
    }
  }
}
```

> На Windows шлях зі зворотними слешами: `"command": "C:\\шлях\\до\\.venv\\Scripts\\mcp-baf.exe"`

Для **Claude Code** — однією командою:

```sh
claude mcp add mcp-baf \
  --env mcp_baf_USER='Адміністратор' \
  --env mcp_baf_PASSWORD='пароль' \
  -- /шлях/до/.venv/bin/mcp-baf \
  --base http://localhost:8080/hs/mcp-baf \
  --dump /шлях/до/вивантаження
```

> Також підтримуються: **Claude Desktop**, **Cursor**, **Windsurf**, **VS Code + Copilot/Continue/Cline**, **JetBrains IDE**, а також будь-які клієнти для локальних моделей з підтримкою MCP.

Перезапустіть AI-клієнт і спитайте: **«Покажи структуру конфігурації моєї бази 1С»**

## Доступні інструменти

| Інструмент | Опис |
|------------|------|
| `get_metadata_tree` | Дерево метаданих: довідники, документи, регістри, загальні модулі та ін. |
| `get_object_structure` | Реквізити, табличні частини, виміри і ресурси конкретного об'єкта |
| `get_form_structure` | Структура форми: елементи, команди, обробники подій (повний склад — із `--dump`) |
| `get_configuration_info` | Ім'я конфігурації, версія, постачальник, версія платформи, режим роботи |
| `search_code` | Повнотекстовий пошук по коду модулів: smart (BM25), regex, exact. BSL-синоніми (рос↔англ). Фільтрація за типом метаданих і модуля. Потребує `--dump` |
| `bsl_syntax_help` | Довідка по 180 вбудованих функціях, методах типів і патернах BSL |
| `execute_query` | Виконати запит мовою запитів 1С з параметрами (тільки SELECT/ВЫБРАТЬ) |
| `validate_query` | Перевірити синтаксис запиту без виконання |
| `get_event_log` | Читання журналу реєстрації з фільтрацією за датою, рівнем і користувачем |

Плюс 11 промптів: `review_module`, `write_posting`, `optimize_query`, `explain_config`, `analyze_error`, `find_duplicates`, `write_report`, `explain_object`, `1c_query_syntax`, `1c_metadata_navigation`, `1c_development_workflow`.

|--------|---------|------------------|------|
| `--base` | `mcp_baf_BASE_URL` | `http://localhost:8080/hs/mcp-baf` | URL HTTP-сервісу 1С |
| `--user` | `mcp_baf_USER` | — | Користувач HTTP-сервісу |
| `--pass` / `--password` | `mcp_baf_PASSWORD` | — | Пароль HTTP-сервісу |
| `--max-response-size` | `mcp_baf_MAX_RESPONSE_SIZE` | `128` | Максимальний розмір відповіді 1С у мебібайтах (MiB). Більша відповідь відхиляється зі зрозумілою помилкою. Збільште ліміт для великих баз із розширеннями |

| `--install` | — | — | Встановити розширення в базу 1С за вказаним шляхом |
| `--lang` | — | `ua` | Мова синонімів розширення для `--install`: `ua` (українська, за замовчуванням) або `ru` (російська) |
| `--server` | — | — | Режим клієнт-серверної бази: `--install` приймає рядок підключення `сервер\база` (наприклад `srv-1c\buh_prod`) |
| `--platform` | — | — | Шлях до бінарника 1С (автовизначення, якщо не вказано) |
| `--platform-version` | — | — | Версія платформи 1С (наприклад `8.3.13`). Визначається автоматично зі шляху до платформи. Вкажіть вручну, якщо платформа в нестандартному шляху без номера версії. Мінімальна підтримувана версія: 8.3.10 |
| `--db-user` | — | — | Користувач бази 1С для DESIGNER (режим `--install`) |

| `--audit-max-size` | `mcp_baf_AUDIT_MAX_SIZE` | `50` | Ротація `audit.log` при перевищенні розміру (MiB) |

## Логування і аудит

Сервер веде два журнали в кеш-каталозі (`--cache-dir`, за замовчуванням платформений кеш). Бізнес-аудит і операційний лог розділені.

### `audit.log` — журнал подій (JSONL)

Єдиний контракт аудиту для сервісів BAF реалізує окремий пакет [`mcp-baf-audit`](https://github.com/andriy4k07/mcp-baf-audit) (його ж використовує `baf-write-mcp`). Кожна подія — старт/стоп сервера, кожен виклик інструмента (`tool_call`/`tool_error` з `tool`, `args`, `duration_ms`), розбіжність версії розширення — пишеться одним JSON-рядком зі спільним конвертом (`schema_version`, `ts`, `service`, `session`, `seq`, `trace_id`). Секрети (`password`/`token`/…) автоматично редагуються до `***`. Журнал ротується за розміром: `audit.log` → `audit-<timestamp>.log`, архіви понад ліміт видаляються (`--audit-max-size`, `--audit-archives`).

### `server.log` — операційний журнал

MCP-клієнти трактують будь-який рядок у stderr як помилку, тому туди потрапляють тільки повідомлення рівня ERROR. Операційний лог завжди пишеться у файл `server.log` на рівні INFO (ротація за розміром, переживає рестарти); `--debug` піднімає рівень до DEBUG і додає подробиці httpx. Прогрес індексації виводиться лише при запуску в терміналі (stdin — tty).

| Прапор / env | Опис |
|--------------|------|
| `--verbose` | Примусово ввімкнути вивід прогресу навіть при запуску через pipe. Зручно для відлагодження підключення MCP-клієнта |
| `--quiet` | Примусово заглушити прогрес навіть у терміналі. Перевизначає `--verbose` |
| `mcp_baf_NO_TTY=1` | Еквівалент `--quiet`. Зручніше за CLI-прапор у Docker / systemd |
| `--debug` | DEBUG-рівень у `server.log` у кеш-каталозі. Також вимикає індикатор прогресу |

> **Git Bash / MSYS2 / MinTTY на Windows:** ці оболонки підключають stdin через іменовані pipes, тому автовизначення вважає їх не-TTY і прогрес не виводиться. Для ручної діагностики використовуйте `--verbose` або повноцінний `cmd.exe` / Windows Terminal.

## Як працює search_code

Пошук працює по локальному вивантаженню конфігурації (`DumpConfigToFiles`). Індекс — SQLite FTS5: при першому старті сервер у фоновому потоці рекурсивно обходить каталог `--dump`, паралельно читає `.bsl` файли і будує індекс; до готовності пошук повертає «search index is building, please retry».

База зберігається в кеш-каталозі (ім'я — sha256 від шляху вивантаження) і на наступних стартах відкривається миттєво: змінені/нові/видалені файли визначаються за mtime+size і застосовуються інкрементально. `--reindex` примусово перебудовує кеш; пошкоджений кеш перебудовується автоматично.

Режими: `smart` (BM25 з двомовними BSL-синонімами), `regex` (синтаксис Python `re` — відмінність від Go-версії, де був Go regexp), `exact` (підрядок без урахування регістру).

## Розробка

```sh
.venv\Scripts\pip install -e ".[dev]"     # встановлення з dev-залежностями
.venv\Scripts\python -m pytest tests      # тести
```

Ручна e2e-перевірка (`scripts/e2e_check.py`) потребує mock-сервер 1С з оригінального Go-репозиторію: `go run ./cmd/mock-1c`.

Довідник BSL-функцій (`src/mcp_baf/bsl/_functions_data.py`) генерується з `bsl/functions.go` Go-версії: `python scripts/gen_bsl_data.py <шлях/до/functions.go>`.

### Розширення 1С

Вихідники розширення зберігаються в `src/mcp_baf/extension_src/` у форматі XML-вивантаження конфігурації і постачаються разом із пакетом. При `--install` вони завантажуються напряму через DESIGNER `/LoadConfigFromFiles` — готовий .cfe файл не потрібен. Перед завантаженням XML патчиться під версію цільової платформи: версія формату вивантаження, режим сумісності, видалення елементів, які не розуміють старі платформи, з ланцюжком повторів під відомі помилки DESIGNER.

## Сумісність

| AI-клієнти | |
|------------|---|
| **Локальні моделі** | Ollama, LM Studio, llama.cpp і будь-які MCP-сумісні клієнти |

|--------------|--------|
| 8.3.10 і вище (комерційна) | Підтримується |
| 8.5.x (комерційна) | Підтримується |
| 8.3.10+ / 8.5.x (навчальна) | Підтримується |

| ОС | MCP-сервер | Автоустановка | HTTP-сервіс 1С |
|----|------------|---------------|----------------|
| Windows | так (Python 3.11+) | так | так (Apache/IIS — рекомендовано, /HTTPPort — для швидкого запуску) |
| macOS | так (Python 3.11+) | так | ні (обмеження платформи 1С), використовуйте Windows-VM |
| Linux | так (Python 3.11+) | так | так (Apache або ibsrv; /HTTPPort недоступний) |
