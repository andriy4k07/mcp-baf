# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

MCP server that connects AI assistants to 1C:Enterprise databases over the 1C HTTP service. Python port of [feenlace/mcp-1c](https://github.com/feenlace/mcp-1c) (Go); source comments frequently reference the corresponding Go files, and tool registration order intentionally matches the Go version.

Code comments and docstrings are in Russian, the README is in Ukrainian — follow that convention in new code.

## Commands

```sh
.venv/bin/pip install -e ".[dev]"          # install with dev deps
.venv/bin/python -m pytest tests           # run all tests
.venv/bin/python -m pytest tests/test_server.py::test_registered_tools  # single test
```

There is no linter or formatter configured. Manual e2e check (`scripts/e2e_check.py`) requires the mock 1C server from the original Go repo (`go run ./cmd/mock-1c`).

The dependency `mcp-baf-audit` is a sibling repo: in local development install it with `pip install -e ../mcp-baf-audit`; releases pin by git tag.

## Architecture

Entry point `__main__.py:main` has two modes: `--install` runs `installer.py` (loads the 1C extension via DESIGNER), otherwise `server.create_server(config)` builds a FastMCP server and runs it on stdio.

**Tool registration pattern.** Each module in `src/mcp_baf/tools/` exposes `register(mcp, client, audit)`; `server.create_server` calls them in a fixed order. Tools return ready-made markdown strings (not JSON). Every tool body is wrapped in `tools/common.py:traced_text`, which emits a `tool_call`/`tool_error` audit event and pins a `trace_id` in a contextvar.

**Audit (mcp-baf-audit, schema v2).** `audit.log` (JSONL, size-rotated) is the business audit; it is separate from the operational `server.log`. `OneCClient` receives the audit writer and emits exactly one `one_c.http` event per 1C call (status, duration, response size — never the body); those events inherit the tool's `trace_id` via the contextvar. Secrets are auto-redacted by the audit package. stderr only gets ERROR-level logs because MCP clients render any stderr line as an error.

**HTTP client (`client.py`).** httpx-based `OneCClient` with basic auth, `Connection: close` (to avoid exhausting 1C session limits), and streamed reads capped at `--max-response-size` so an oversized response fails with a clear `OneCError` instead of OOM.

**Search index (`dumpindex/`).** SQLite FTS5 index over `.bsl` files from a `--dump` config export. Built in a background thread — the server starts immediately and `search_code` returns "index is building" until ready. Disk cache in the cache dir (name = sha256 of dump path) with incremental updates by mtime+size; SQLite access is serialized with a lock. `smart` mode expands query tokens with ru↔en BSL synonyms (`synonyms.py`) before BM25 ranking; `regex`/`exact` scan module contents line by line. Without `--dump`, `search_code` is not registered at all (tests rely on this).

**1C extension (`extension_src/`).** XML config-dump sources of the extension, shipped inside the package. `installer.py` copies them to a temp dir, patches the XML for the target platform version (dump format version, compatibility mode, elements unknown to old platforms) and loads via DESIGNER `/LoadConfigFromFiles` with a retry chain for known errors. `server.py:EXPECTED_EXTENSION_VERSION` must match the extension version — the server checks `/version` at startup and logs/audits a mismatch. When changing `extension_src/`, bump both.

**Config (`config.py`).** Precedence: defaults < env vars (prefix `mcp_baf_`) < CLI flags. The service URL path `/hs/mcp-baf` is hard-coded in the extension's `RootURL` and must match `--base`.

**Generated code.** `src/mcp_baf/bsl/_functions_data.py` (BSL reference for `bsl_syntax_help`) is generated by `python scripts/gen_bsl_data.py <path/to/functions.go>` from the Go repo — don't edit it by hand.

## Development workflow (owner's hard rules)

- **Never open a PR before the owner has tested the change on the live 1C base himself.** Loop: implement → he reinstalls the extension/package on the live base → he says "перевіряй" → verify through the MCP tools → then branch/commit/PR.
- Branch per task named after the actual work (`fix/<what>`, `feature/<N>-<name>`); commit in logical blocks (several commits grouped by file/feature); commit messages in English; PR title in English (body may be Ukrainian). He often asks for the PR title+description as markdown text to paste himself.
- Only the owner can install the extension on the live base (`--install`; supports `--lang ua|ru`, default Ukrainian — the target base's primary language is Ukrainian). Remember the `EXPECTED_EXTENSION_VERSION` sync rule above on every `extension_src/` change.
- Ecosystem context: this is the read side; the write side is the sibling `baf-write-mcp`, shared logging is `mcp-baf-audit` (PyPI), and both feed `baf-ops-dashboard` and the remote hermes-agent. Everything is branded "baf" — don't reintroduce "1c" into URLs, names, or docs.
